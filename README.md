
# Automatically Resize Images Using AWS S3, Lambda, SNS, and Terraform

Image processing is a common requirement in modern applications. Whether you're building an e-commerce platform, a portfolio website, or a social media application, users often upload images that need to be resized before being stored or displayed.

Manually resizing images is not practical. It consumes time, increases operational effort, and becomes difficult to manage as the number of uploads grows.

A better approach is to automate the entire workflow.

In this article, we'll build a fully automated, serverless image processing pipeline using AWS and Terraform. Whenever a user uploads an image to Amazon S3, AWS Lambda automatically resizes it, stores the processed image in another S3 bucket,  and sends an email notification through Amazon SNS, and records execution logs in CloudWatch.

The entire infrastructure is provisioned using Terraform.

---

# Business Problem

Many organizations allow users to upload images such as profile pictures, product photos, or marketing assets.

These uploaded images are often much larger than required.

Large images increase storage costs, consume additional bandwidth, and negatively impact website performance.

As the application grows, manually processing every uploaded image becomes impossible.

Companies need an automated solution that processes images immediately after upload without requiring servers or manual intervention.

---

# Solution

In this project, we build a serverless event-driven architecture.

Whenever an image is uploaded to an Amazon S3 bucket:

* Amazon S3 triggers an AWS Lambda function.
* Lambda resizes the uploaded image using the Pillow library.
* The resized image is stored in a separate S3 bucket.
* Amazon SNS sends an email notification confirming successful processing.
* CloudWatch Logs record the execution details.

All AWS resources are created using Terraform.

# Step 1 — Deploy the Infrastructure

Run the following Terraform commands.

```bash
terraform init

terraform plan

terraform apply
```

During deployment, make the following changes according to your environment.

* Update the AWS Region if you are deploying outside **ap-south-1**.
* Replace the SNS email address with your own email address.
* If you deploy to another AWS Region, update the Pillow Lambda Layer ARN because Lambda Layers are region-specific.

After reviewing the execution plan, enter:

```
yes
```

Terraform creates:

* Source S3 Bucket
* Resized Images Bucket
* Lambda Function
* IAM Role
* IAM Policies
* SNS Topic
* SNS Subscription
* Lambda Permission
* Bucket Notification

## Verify the Lambda Function

After running `terraform apply`, open the **AWS Lambda Console**.

You should see the newly created Lambda function.

Review the following configuration:

#### Function Configuration

Verify that the Lambda function has been created successfully and is using the expected Python runtime. You can also review the configured memory size and timeout values defined in Terraform.

#### Environment Variables

Open the **Configuration → Environment variables** section.

You should see the following variables:

* `DESTINATION_BUCKET_NAME` – Specifies the S3 bucket where the resized images will be uploaded.
* `SNS_TOPIC_ARN` – The Amazon SNS topic used to send email notifications after image processing.
* `RESIZE_PERCENTAGE` – Controls how much the image is resized (for example, `50` resizes the image to 50% of its original dimensions).
* `JPEG_QUALITY` – Defines the compression quality of the output JPEG image. Lower values reduce file size, while higher values preserve image quality.

Using environment variables makes the Lambda function more flexible because configuration values can be changed without modifying or redeploying the application code.

#### Trigger

Navigate to the **Triggers** section.

You should see that the **Source S3 Bucket** is configured as an event source. Whenever a new `.jpg` image is uploaded, Amazon S3 automatically invokes the Lambda function to process the image.

#### IAM Execution Role

Open the **Permissions** tab and review the execution role attached to the Lambda function.

This IAM role grants the minimum permissions required for the function to operate, including:

* Read images from the source S3 bucket.
* Upload resized images to the destination S3 bucket.
* Publish notifications to Amazon SNS.
* Write logs to Amazon CloudWatch Logs.

These permissions are automatically created and attached by Terraform.

## Clean Up Resources

To avoid unnecessary AWS charges, delete all resources after completing the lab.

**Important:** Terraform cannot delete an S3 bucket unless it is empty.

Before running `terraform destroy`:

1. Open the **Source S3 Bucket** and delete all uploaded images.
2. Open the **Resized S3 Bucket** and delete all processed images.
3. Empty both buckets completely.

Once both buckets are empty, run:

```bash
terraform destroy
```

Review the execution plan, then type:

```text
yes
```

Terraform will remove all AWS resources created during this project, including:

* S3 Buckets
* Lambda Function
* IAM Role and Policy
* SNS Topic and Email Subscription
* Lambda Permission
* S3 Bucket Notification

This ensures your AWS account is left clean and helps prevent unnecessary charges.

> **Note:** If `terraform destroy` fails with a `BucketNotEmpty` error, verify that both S3 buckets have been completely emptied, then run the command again.

# Best Practices

Although this project is designed as a learning lab, several improvements can make it production-ready.

* Use S3 Versioning.
* Enable Bucket Encryption.
* Enable Dead Letter Queues (DLQ).
* Add Lambda retries.
* Validate uploaded file types.
* Support multiple image sizes.
* Configure AWS Budgets to monitor costs.
* Apply least-privilege IAM permissions.
